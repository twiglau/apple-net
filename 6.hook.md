# Hook 使用

## 常见 Hook

```lua
Hook 名称            作用说明
useState            用于在函数组件中添加状态。
useEffect           用于在函数组件中添加副作用。
useContext          用于在函数组件中访问上下文。
useReducer          用于在函数组件中管理状态,管理复杂状态逻辑的替代方案。
useCallback         用于在函数组件中缓存函数（记忆化的回调函数）,避免不必要的重新渲染。
useMemo             用于在函数组件中缓存计算结果（记忆化的计算结果）。
useRef              用于在函数组件中访问 DOM 元素，或者保存不触发重新渲染的值。
useImperativeHandle 用于自定义暴露给父组件的实例值。
useLayoutEffect    用于在函数组件中添加副作用，与 useEffect 类似，但会在 DOM 更新后同步调用。
```

## 设计哲学

- 逻辑复用： 通过自定义 Hook, 可以将组件的逻辑解耦，使得我们能够复用逻辑，而不是在多个组件中重复相同的逻辑。
- 函数式编程： Hook 提供了函数式编程的思维方式，使得我们能够以函数的形式组织代码，而不是以类的形式组织代码。
- 状态与 UI 分离： 在 React 中，状态和渲染逻辑是解耦的。通过 Hooks,我们能够在函数组件中独立管理状态，而不需要通过类组件中的生命周期方法来做。

## 核心应用规则

1. 只在顶层调用 Hooks
2. 仅在 React 函数中调用 Hooks

## Hook 命名规范

- 按照 React 的约定所有的 Hooks 需要以 use 开头,返回一个 state 状态对象，并且可以调用
  其他 Hook,同时遵循组合式设计。

- React 在开发环境中，使用 StrictMode 进行双重调用，帮助我们检测内存泄漏等问题；
- 同时，在一些特殊场景中，如依赖关系不当，会触发循环引用。导致无限渲染。

```jsx
function DangerousComponent() {
  const [state, setState] = useState(0);
  useEffect(() => {
    setState((prev) => prev + 1); // 可能触发无限循环
  }, [state]); // 危险依赖
}
```

## 最佳实践总结

1. 纯净性原则： 保持 Hooks 的输入，输出是可预测的。
2. 原子化设计： 每个 Hook 只专注于一个功能，避免做多余的事。
3. 依赖显示生命：使用 useEffect 等 Hooks 时，确保依赖项的声明是完整的。
4. 引用稳定性： 通过 useCallback,useMemo 等 Hooks 保持引用的稳定性。避免不必要的重新渲染。
5. 分层架构： 将业务逻辑和 UI 组件分离，使得代码更具可维护性。

## 闭包风险

- 什么是闭包？

1.  一个函数能 ”记住“ 它定义时的作用域，即使这个函数到别的地方执行了，它也能访问原来那个作用域里的变量。
2.  闭包 = 函数 + 外部变量的 ”打包组合“

- 闭包冻结问题是什么？
  闭包冻结，就是函数内部引用的状态值，不是 ”最新的值“，而是 ”它被定义那一刻“ 的快照。

- 这在 React 里尤其常见，因为 React 的状态更新是异步的，每次渲染都会生成一个新的 ”闭包版本“，
  之前的闭包不会自动知道状态变了。

## 严格模式： Hook 运行两次？

- StrictMode 是 React 开发模式下的一个工具，它的目标是帮助我们提前发现潜在的 Bug,特别是一些和
  副作用，异步操作，组件生命周期相关的问题。它会在开发环境（注意，不是生产环境！）下执行一些额外的
  检查，其中最”坑“的一个特性就是 -- 双重调用组件的函数体。
- 假如你组件挂载后，立马因为某种原因（比如路由切换，状态变化）被卸载，再快速重新挂载，它还能正常运行吗？
- 通过这种 ”快速挂载” + “清理” + “再挂载” 的流程，React 能检测出你代码里有没有写漏掉的地方，比如：

1. 有没有忘记取消订阅
2. 有没有遗留的定时器或网络请求
3. 有没有错误地更新了已经卸载的组件状态

- React 为什么要这样设计？

1. 检查你有没有写对 useEffect 清理逻辑
2. 逼你写出 “幂等”的副作用代码
   > React 的副作用必须是幂等的(idempotent),意思就是：你无论执行几次，结果都得一样，不能炸锅。
   > 比如你不能在副作用里直接依赖外部状态，或者执行“只允许一次”的初始化逻辑。
3. 为未来的并发模式打基础。

## 无效渲染： useCallback + React.memo

## useMemo 应用

- 如果你不加 useMemo,每次组件一更新， .filter() 就会重复执行，不管 searchResults 有没有变，浪费性能。
- useMemo 是提升性能的利器，但也别滥用。什么时候该用？

  > 当你的计算开销比较大 （比如 .filter(), .map(), 排序，大数组处理等）
  > 并且依赖项明确，变化频率不高的时候，用它就对了。

## 例子

在 SearchResults.tsx 中使用 useMemo 是合适的，这与之前在 ToggleButtons.jsx 中建议 不要 使用的情况有本质区别。可以使用以下三点来理解：

1. 计算成本不同（性能）

- ToggleButtons: 只是简单的字符串拼接（px-3... + selected）。这种操作在 JS 中极其快，几乎不占 CPU。
- SearchResults: 这里涉及到了数组的 .filter() 操作。
  - 虽然目前数据量可能不大，但 searchResults 是从 API 获取的，可能会有几十甚至上百条数据。
  - 如果没有 useMemo，组件内任何微小的状态变化（比如你正在输入搜索框、或者点击了页面上的其他开关）都会导致这个 .filter() 重新跑一遍。

2. 引用稳定性（关键点），这是最重要的原因。

- 在 ToggleButtons 中，返回的是 字符串（基本类型）。字符串只要内容一样，React 就认为它没变。
- 在 SearchResults 中，.filter() 每次执行都会返回一个 全新的数组对象（引用地址变了）。
  - 你的 SearchProductCard 组件使用了 React.memo 进行了优化。
  - 如果不用 useMemo，即便过滤结果完全没变，因为 displayProducts 的数组引用变了，React 会认为数据更新了，从而导致 所有的商品卡片（SearchProductCard）全部强制重新渲染。
  - 使用 useMemo 后，只要 searchResults 和 selectedFilter 没变，displayProducts 的引用就保持不变，React.memo 就能生效，跳过不必要的子组件渲染。

3. 语义化与代码结构
   - 将过滤逻辑从“渲染时调用的函数”变成一个“根据依赖变化的变量”，更符合 React 的声明式编程习惯。
   - 它明确告诉阅读代码的人：这个变量只在数据源或过滤条件改变时才会更新。

总结对比图
特性 | ToggleButtons (建议不用) | SearchResults (建议使用)
--- | --- | ---
返回类型 | 字符串 (基本类型) | 数组 (对象引用)
计算复杂度 | 极低 (拼接) | 中等 (遍历过滤)
对子组件影响 | 无 | 影响较大的子组件树是否重绘
优化目的 | 只有性能 (在这里被负优化) | 性能 + 引用稳定性 (配合 React.memo)

一句话总结：当计算结果是对象或数组，且该结果会被用于 .map() 渲染子组件 或作为 Props 传给 memo 组件 时，useMemo 是非常有价值的。

## useRef - 引用持久化与 DOM 的操作逻辑

- useRef() 是访问 DOM 的推荐方式，比 querySelector 更安全，更 React 化。
- 它不会触发重新渲染，性能也更高效。
- ref.current 属性指向的就是你想要的真实 DOM.
- 推荐在 useEffect 中使用它，确保组件挂载后，再操作 DOM.
