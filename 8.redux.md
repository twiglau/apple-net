## 从 Thunk 到 RTK Query

- RTK Query 简化异步请求
- 减少 80% 模版代码
- 提升可维护性

## 创建 searchAPi 服务

1. 使用 createApi + fetchBaseQuery 创建 API 服务
2. 定义 endpoints => 自动生成 useSearchQuery
3. 自动处理缓存，加载，错误

5. 挂载 searchApi.reducer, 加入 searchApi.middleware

- 可能类型： FetchBaseQueryError | SerializedError | undefined
- 使用 getErrorMessage 统一处理
- 避免 UI 显示 【object object】

## 语言切换方案

### 前端动态加载

```js
// 语言状态保存在组件内
const [lang, setLang] = useState("zh");

// 切换时重新加载语言包
const changeLanguage = (newLang) => {
  fetch(`/locales/${newLang}.json`)
    .then((res) => res.json())
    .then((data) => {
      setTranslations(data);
      setLang(newLang);
    });
};
```

- 状态丢失
- 传递频繁
- 页面切换反复初始化

### 方案三： Context API

```js
// 创建语言上下文
const LanguageContext = createContext();

// 根组件提供值
<LanguageContext.Provider value={{ lang, changeLanguage }}>
  <App />
</LanguageContext.Provider>;

// 子组件消费
const { lang } = useContext(LanguageContext);
```

- 性能拉胯
- 难以数据持久化
- 无法多标签同步

### Redux 解决方案

- Redux 就是一个中英语言控制中心： 全局只有一个大脑，所有组件都能随时接入，不会丢状态，不会乱传值。

  > 1. 全局共享
  > 2. 缓存优化
  > 3. 持久化存储
  > 4. 多标签页同步

- Redux 的设计逻辑： 死板，但优雅

  > View -> Action -> Reducer -> Store -> View

- Reducer (状态更新逻辑)

  > “流水线上的机器”，它是一个纯函数（Pure Function).
  > 输入一样，输出永远一样(没有随机数，没有 API 调用).
  > Reducer 就像一台按规矩办事的机器人，给它一个旧状态 + Action, 它就吐出一个新状态。Store 本身 是只读的，任何人都不能直接去动它，只能交给 Reducer 来处理。

- Action (状态变更指令)
  > 像是 “指令单“ 或者 ”快递单”。
  > 它不做任何逻辑运算，只描述 “我要做什么”。
  > 格式页超级简单，最经典的就是 {type: "INCREMENT"}.
  > 所有状态的变化都能被完整追踪和记录。配合 Redux DevTools, 你甚至能做到时间旅行（Time Travel Dubugging): 回溯到某个历史状态，就像给应用按了个 “倒带键”。

### Redux Toolkit

- 在传统 Redux 里，你得手动写 action, 写 reducer, 还得自己配置 store. 处理异步逻辑还要搭配各式各样的中间件。
- Redux Toolkit 做了减法：

```lua
传统 Redux                     Redux Toolkit
手动写 action + reducer         自动生成reducer
手动配置 store                   一行代码创建 store
手动处理异步                      内建 createAsyncThunk
容易写错                         TypeScript支持很好
自由度高，但难维护                 opinionated (有官方推荐方式)
```

- Redux Toolkit 架构

1. 在传统 Redux 里，写一个功能要分成好几块： state, actions, reducer
2. RTK 提供了一种全新的写法： Slice 架构。
   - 每一块状态配上自己的小工具箱，状态 + 修改方法都收在一起，不用再分文件。
   - 状态和逻辑在同一个地方，高内聚；
   - 每个 slice 切片都能独立管理，互不干扰，低耦合；
   - 更直观，更好维护；

## 对比传统的 Redux, 传统 Redux 的问题

- 手写字符串容易翻车： "SET_CULTURE" 拼错， TS 不一定能发现。
- reducer 可读性差： 全靠 switch...case. 大堆 if, 读起来费眼。
- 状态更新得手动返回新对象： 每次都写 { ...state, xxx: yyy}.
- 样板代码爆炸： 一个功能需要写三份： action type, action creator, reducer.

## RTK 优势

- 写法更直观： 看似 “直接改动 state", 其实底层用 immer 自动保证不可变。
- 不用写 action type: createSlice 自动帮你生成，避免拼写错误。
- 类型安全感拉满： PayloadAction<CultureCode> 直接限制 payload 类型。
- 导出一把梭： action 和 reducer 一起同时生成，代码清爽。
